=== Description of the Bug and Fix ===

The original btranslate.cpp pintool (Probe mode) failed with a Segmentation
fault when run on cpugcc_r_base.Oz-m64, producing no 200.s output.

Five bugs were found and fixed:

Bug 1 - Off-by-one error in jump_to_orig_addr_map (fix_direct_jmp_or_call_to_orig_addr):
  The original code incremented jump_to_orig_addr_num BEFORE assigning it as
  the new entry index, so entries started at index 1 (index 0 was never used).
  The deduplication search looped from 0 to jump_to_orig_addr_num-1, which
  never covered the actual stored entries, so every call added a new slot
  instead of reusing existing ones. This caused the map to fill up much faster
  than its max_rtn_count capacity.
  Fix: assign the index first, store the entry, then increment the counter.

Bug 2 - Missing RTN_IsSafeForProbedReplacement check (commit_translated_rtns_to_tc):
  The original code called RTN_ReplaceProbed for every translated routine
  without first verifying that the routine is safe for probe insertion. For
  routines that are too small (fewer than 5 bytes) or that share code with
  adjacent routines, writing the probe trampoline overwrites neighbouring code,
  causing memory corruption and a subsequent Segmentation fault.
  Fix: call RTN_IsSafeForProbedReplacement(rtn) and skip the routine if it
  returns false.

Bug 3 - Double replacement of _exit causes infinite recursion (create_tc):
  The create_tc callback runs for every image loaded (libc, the main binary,
  and any other shared library). RTN_FindByName(img, "_exit") can return a
  valid routine in more than one image: the real _exit in libc and the PLT
  stub for _exit in the main binary. The original code replaced _exit without
  checking whether it had already been replaced. The second call to
  RTN_ReplaceProbed overwrote origExit with a trampoline pointing back to the
  PLT stub, which is itself probed to call ExitInProbeMode. When the program
  eventually called _exit, ExitInProbeMode called (*origExit), which jumped
  back into ExitInProbeMode via the PLT probe, creating infinite recursion,
  a stack overflow, and a Segmentation fault.
  Fix: guard the replacement with origExit == NULL so _exit is probed only
  once (the first valid, safe occurrence found across all images). Also added
  a NULL check before calling (*origExit) in ExitInProbeMode as a safety net.

Bug 4 - Wrong call order in fix_rip_displacement corrupts RIP-relative encoding:
  In fix_rip_displacement, xed_encoder_request_set_memory_displacement was
  called BEFORE xed_encoder_request_init_from_decode. The init_from_decode
  call copies all fields from the decoded instruction into the encoder request,
  which OVERWRITES the corrected displacement that was just set. As a result,
  every RIP-relative instruction in the translation cache was re-encoded with
  its original displacement rather than the recalculated one. This caused
  incorrect memory accesses in the TC — most went unnoticed because the TC is
  allocated close to the original code, but eventually an access landed in the
  wrong segment, producing a Segmentation fault during execution.
  Fix: call xed_encoder_request_init_from_decode first, then call
  xed_encoder_request_set_memory_displacement to override the displacement.
  (This matches the correct pattern already used in fix_direct_jmp_or_call_
  displacement, which calls init_from_decode before set_branch_displacement.)

Bug 5 - Translating PLT stubs breaks lazy dynamic-symbol resolution (find_candidate_rtns_for_tc):
  After the first four fixes the tool still crashed with a Segmentation fault.
  Isolating the crash with the -limit_rtn knob showed that the failure was
  non-monotonic and layout-dependent (it appeared for some routine counts and
  not others), the classic signature of memory/stack corruption that only
  sometimes lands fatally. A core-dump backtrace pinned the crash inside
  _dl_fixup / _dl_runtime_resolve_xsavec (the dynamic linker's lazy PLT
  resolver), with a garbage link_map pointer.
  The cause: the candidate-selection loop was translating the PLT stub routines
  (entries named "<sym>@plt") and relocating them into the translation cache.
  A PLT stub implements lazy binding: on first call it pushes a relocation index
  and jumps to the resolver, which depends on the stub's exact original location
  and on the stack alignment at entry (xsavec needs an aligned save area). Moving
  the stub into the TC corrupts the resolver's arguments and the saved-state
  buffer, crashing inside _dl_fixup.
  Fix: skip any routine whose name ends with "@plt" in
  find_candidate_rtns_for_tc, leaving PLT stubs as original code. Calls from
  translated routines still reach functions correctly through the original PLT.
  With this fix the full binary is translated successfully and 200.s is produced
  byte-for-byte identical to the reference output.

A -limit_rtn KNOB was also added to support binary-search isolation of faulty
routines during debugging (pass -limit_rtn N to translate only the first N
routines). The _init routine is also skipped from translation as its translated
code causes a run-time failure.

=== Testing ===

The tool was built with Pin 4.0 (pin-4.0-99633) on Linux x86-64 and run with the
exact command below. It completes with exit code 0 and produces 200.s identical
(byte-for-byte / matching SHA-256) to the provided reference 200.s, across
repeated runs.

=== Compilation ===

Set PIN_ROOT to your Pin 4.0 installation directory, then run:

    make PIN_ROOT=/path/to/pin

The compiled tool will be placed at:

    obj-intel64/btranslate.so

The submitted binary ex3.so is exactly this compiled obj-intel64/btranslate.so
(renamed to ex3.so as required by the submission instructions).

=== How to Run the Tool ===

Using the submitted prebuilt binary:

    $PIN_ROOT/pin -t ex3.so -- cpugcc_r_base.Oz-m64 200.i -o 200.s

Or using the freshly compiled binary:

    $PIN_ROOT/pin -t obj-intel64/btranslate.so -- cpugcc_r_base.Oz-m64 200.i -o 200.s

Output: 200.s (assembly translation of the 200.i input file)
